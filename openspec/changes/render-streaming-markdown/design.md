## Context

Currently, `ChatView.tsx` renders streaming responses in a monospace text container (`<div className="font-mono whitespace-pre-wrap">`) to avoid structural markdown issues. This document outlines how to safely render streaming responses using `ReactMarkdown` with preprocessed auto-closed markdown tokens.

## Goals / Non-Goals

**Goals:**
- Render streaming text as Markdown dynamically as it is generated by the AI agent.
- Automatically detect and close unclosed code blocks (` ``` `) to prevent visual layout breakage.
- Maintain visually identical custom rendering of code blocks (including Copy, Insert, and Replace buttons) and other Markdown formatting for streaming text as for completed messages.
- Render a blinking cursor at the end of the streaming Markdown output.

**Non-Goals:**
- Auto-closing other complex HTML or nested markdown structures beyond standard markdown syntax (focusing primarily on code blocks, which are the main source of breakage).

## Decisions

### 1. Preprocessor Location and Logic
We will create a new utility file at `src/sidepanel/utils/markdown.ts` containing the preprocessing function `preprocessStreamingMarkdown(text: string): string`.
- **Logic**: Count instances of "```" in the text. If the count is odd, it indicates an open block. We will append `\n```` to close it.
- **Why**: Keeps UI code clean and makes unit testing the preprocessor simple and isolated.

### 2. Sharing Markdown Components
We will refactor the custom `markdownComponents` definition from `MessageBubble.tsx` into a shared React component or a reusable helper function.
- **Option A**: Duplicate the components map inside `ChatView.tsx`.
- **Option B**: Move `markdownComponents` definition and associated action handlers (copy, insert, replace) to a shared file/hook.
- **Decision**: We will refactor/extract the markdown rendering configuration to a shared hook or file so that both `MessageBubble.tsx` and `ChatView.tsx` render code blocks and text consistently.

### 3. Cursor Indicator Placement
Instead of a separate absolute block or absolute-positioned cursor, we will append a custom cursor symbol (like a pulsing vertical bar `▋`) directly to the preprocessed markdown string before rendering, or append it to the last text node to prevent line breaks.
- **Why**: Appending the cursor to the text nodes directly prevents the cursor from jumping to a new line on its own unless the text itself has a trailing newline.

## Risks / Trade-offs

- **[Risk] Preprocessing delays or CPU overhead** → **Mitigation**: The regex match and backtick counting are extremely fast for text under 100k characters. We will ensure the implementation is lightweight.
- **[Risk] Cursor breaks markdown parsing** → **Mitigation**: If we append the cursor inside an open code block, it will be rendered inside the code block. If we append it outside, it will be rendered outside. The preprocessor must handle appending the cursor *before* auto-closing the code block to ensure the cursor stays within the code block if it is currently typing inside it.
